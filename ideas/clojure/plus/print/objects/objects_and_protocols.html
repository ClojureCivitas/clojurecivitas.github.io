Printing Objects in Clojure

The Clojure default for printing objects is noisy:

(Object.)
#object[java.lang.Object 0x36f1f8fd "java.lang.Object@36f1f8fd"]

The syntax is #object[CLASS-NAME HASH toString())] and as you can see, the toString of an Object is CLASS-NAME@HASH. This can get pretty ugly:

(async/chan)
#object[clojure.core.async.impl.channels.ManyToManyChannel 0x3b4069df "clojure.core.async.impl.channels.ManyToManyChannel@3b4069df"]

clojure-plus provides print-methods to improve printing many things.

(require 'clojure+.print)
^:kind/hidden
(clojure+.print/install-printers!)

Once activated, we can print functions, atoms, namespaces, and more sensibly

inc
#fn clojure.core/inc
(atom {})
#atom {}
*ns*
#ns clojure.plus.print.objects.objects-and-protocols

Clojure Plus adds printers for many types, but no printer is provided for Object, which remains as Clojureâ€™s default printing method. There are plenty of objects left over that print messily. Going back to our channel example, it still prints the same:

(async/chan)
#object[clojure.core.async.impl.channels.ManyToManyChannel 0x77287c8b "clojure.core.async.impl.channels.ManyToManyChannel@77287c8b"]

Itâ€™s not hard to provide an Object print-method:

(defmethod print-method Object [x ^java.io.Writer w]
  (.write w "#object ")
  (.write w (.getName (class x))))
#multifn print-method
(async/chan)
#object clojure.core.async.impl.channels.ManyToManyChannel

Much nicer! In my opinion this is a big improvement. Especially in the world of notebooks where we like to show things as we go, but also just keeping a tidy REPL or looking into data that contains objects.

Hmmmm. not so nice. Weâ€™ll dig into this further below. But we also need to be aware that Clojure munges itâ€™s names to make Java valid names. This matters for some things:

Whoa, thatâ€™s pretty gross. Weâ€™d prefer to demunge the names at least.

(defn class-name
  [x]
  (-> x class .getName Compiler/demunge))
(-> ((fn %% [] (fn %%% [])))
    (class-name))
"clojure.plus.print.objects.objects-and-protocols/eval47397/%%--47398/%%%--47399"

Notice the /evalNNNNN/ part? To create a function, Clojure creates a new class. The /evalNNNNN/ counts every time it evaluates. This is useful in the sense that it identifies the class for that evaluation. But we almost never care for that detail (more on that later). For the same reason our strangely named functions have --NNNNN appended to them, because they are sub evaluations of the top-level evaluation.

Letâ€™s do away with that noise for the moment:

(defn remove-extraneous
  "Clojure compiles with unique names that include things like `/eval32352/` and `--4321`.
  These are rarely useful when printing a function.
  They can still be accessed via (class x) or similar."
  [s]
  (-> s
      (str/replace #"/eval\d+/" "/")
      (str/replace #"--\d+(/|$)" "$1")))
(-> ((fn %% [] (fn %%% [])))
    (class-name)
    (remove-extraneous))
"clojure.plus.print.objects.objects-and-protocols/%%/%%%"

Looking better, I can actually see the (strange) name of the functions.

Notice now that the string is not a valid symbol? Thatâ€™s a problem if we follow the style of clojure-plus because it prints as tagged literals.

#object clojure.plus.print.objects.objects-and-protocols/%%/%%% is not a valid Clojure expression. In a sense it doesnâ€™t matter much if we are just printing, but it tends to confuse tools (and people). So I recommend replacing all slashes after the namespace delimiter with $ instead. While we are at itâ€¦ the namespace isnâ€™t helping us at all here. Letâ€™s just show the name without the namespace.

(defn format-class-name [s]
  (let [[ns-str & names] (-> (remove-extraneous s)
                             (str/split #"/"))]
    (if (and ns-str names)
      (str/join "$" names)
      s)))
(-> (((fn aaa [] (fn bbb [] (fn ccc [])))))
    (class-name)
    (format-class-name))
"aaa$bbb$ccc"

Thatâ€™s really all we need. Notice there is no / slash if we donâ€™t show the namespace, but we still canâ€™t use / because functions can nest arbitrarily.

Side note about including or omitting the namespace. There are many ways to get whatever representation you want, so changing the printing behavior doesnâ€™t prevent anything. The question is just would you like to see it or not? I would argue that itâ€™s almost never useful, just a distraction. On the other hand, the current behavior for printing vars is fully qualified. Whatever your preference on showing the namespace, it should probably be consistent across printers.

Iâ€™ll assume you prefer including the namespace

(defmethod print-method Object [x ^java.io.Writer w]
  (let [class-name (.getName (class x))
